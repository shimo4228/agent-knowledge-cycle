# ADR-0002: Immutable Episode Log as Source of Truth

## Status
accepted

## Date
2026-04-09

## Context

A self-improving agent needs a trustworthy record of what it did, saw, and
decided. If the record is mutable, distillation becomes unreproducible: the
same past cannot yield the same present. If the record is scattered across
structured stores (databases, summaries, caches), the canonical history
diverges from the derived views.

## Decision

The **episode log** is the single source of truth for everything the agent
experiences. All higher layers (knowledge, rules, identity) are pure functions
of the episode log plus a configuration template.

The episode log is:

- **Append-only** — records are never rewritten or deleted in normal operation.
- **Line-delimited JSON (JSONL)** — one event per line, self-contained.
- **Daily-partitioned** — one file per UTC day (`YYYY-MM-DD.jsonl`).
- **Owner-readable only** — written with `umask(0o177)` so the file is created
  with mode `0600`. The log may contain sensitive episode content and is
  treated as a local-only artifact.
- **Schema-versioned** — each record follows `{ts, type, data}` (see
  [`schemas/episode-log.schema.json`](../../schemas/episode-log.schema.json)).

### Record shape

```json
{"ts": "2026-04-09T12:34:56+00:00", "type": "session", "data": {...}}
```

| Field | Meaning |
|-------|---------|
| `ts` | ISO-8601 UTC timestamp, assigned at write time |
| `type` | Event category (e.g. `session`, `action`, `observation`, `insight`) |
| `data` | Arbitrary JSON payload, owned by the adapter that wrote it |

### Consequences for the rest of the harness

- Knowledge and rules are **derived artifacts**. They can be regenerated by
  re-running distillation over the same log.
- Running the same distillation twice on the same log and template must
  produce equivalent output. Non-determinism is the LLM's job, not the log's.
- Privacy is local: the log never leaves the host by default. Adapters that
  want to publish derived state do so explicitly through a separate channel
  governed by ADR-0006.

## Alternatives Considered

- **Mutable structured store (SQLite, Postgres)** — Richer queries, but the
  canonical history can diverge from derived views, and migrations silently
  rewrite the past.
- **Event store with compaction** — Compaction destroys the property we want:
  that the same inputs always reproduce the same outputs.
- **Single large log file** — Simpler, but growth becomes a bottleneck and
  daily partitioning is nearly free to add.

## Consequences

- Disk usage grows monotonically. Operators decide retention by archiving or
  deleting old day-files; the harness does not truncate on its own.
- Any tool that wants to understand "what has happened" reads the log. There
  is no secondary summary to keep in sync.
- Debugging is reproducible: save the log, re-run the distillation, compare.

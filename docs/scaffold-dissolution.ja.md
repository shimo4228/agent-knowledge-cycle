# Scaffold Dissolution — AKC スキルの自己消滅性

## Concept

AKC のスキル群は **足場 (scaffold)** である。サイクルを回すうちにユーザーと AI の両方がサイクルを理解し、対話の中で自然に必要なことを行うようになる。やがてスキルを明示的に呼ぶ必要がなくなる。

```
Phase 1: スキルが行動を教える
  /skill-stocktake → 棚卸しの手順を知る
  /skill-comply → 遵守率を計測する方法を知る

Phase 2: サイクルが身体化される
  スキルを呼ばなくても、対話の中で棚卸し・遵守チェックが自然に起きる
  ルール (testing.md 等) とメモリ (feedback 等) にサイクルの原則が蓄積される

Phase 3: スキルは不要になる
  ユーザー: サイクルの思考パターンが定着
  AI: ルールとメモリからサイクルの原則を再構成できる
  スキルという形式化が冗長になる
```

## Evidence

### 2026-03-29 — スキルなしでサイクルが回った

2026-03-29 のセッションで、AKC の全フェーズが対話的に実行された。どのスキルも明示的に呼ばれなかった:

| AKC スキル | フェーズ | 対話で行われたこと |
|-----------|---------|-----------------|
| search-first | Research | ECC v1.9.0 の差分を調べ、Python 関連を評価 |
| skill-stocktake | Curate | 33 スキルを 1 個ずつ対話で判定 → 16 に削減 |
| skill-comply | Measure | 「eval-harness は testing.md と重複」と発見 |
| rules-distill | Promote | 整理の中で feedback_python_review を memory に保存 |
| learn-eval | Extract | 2 つの記事ドラフトとして知見を外部化 |
| context-sync | Maintain | /context-sync を最後に実行し整合性を確認 |

### 2026-06-10 — プラットフォーム吸収による溶解

3 ヶ月後、当初の概念が想定していなかった第二の溶解経路が観測された。個人ハーネス (Claude Code、ECC エコシステム上に構築) の棚卸しで、ホストプラットフォームが手組みの検証ループのネイティブ版を出荷していたことが判明した: built-in の `verify` / `code-review` / `security-review` skill と native な plan mode である。わずか 3 ヶ月前に工夫を重ねて組んだローカル版は、名前衝突する重複物になっていた。

結果: 6 skill と 1 rule ファイルを削除 (さらに stale な節を trim)、正味 −1,429 / +124 行。明示的な退役基準 (built-in 重複 / モデル知識による自明化 / バージョン依存記述) とともにハーネス側の ADR に記録された。棚卸し自体は素朴な会話のリクエストから始まり、`/skill-stocktake` を呼ぶことなく Curate フェーズが最後まで走った — 2026-03-29 の観測の再現である。

棚卸しを生き残ったものは明快な法則に従う (後述「What Survives」参照): 453 行の OWASP チェックリスト (モデル知識との差分ゼロ) は退役し、45 行の個人用セッション起動 skill (差分 100%) は残った。

## Why This Happens

### ユーザー側の学習

AKC のサイクルを繰り返すうちに、「棚卸しすべきタイミング」「蒸留すべきパターン」「計測すべき指標」を暗黙的に理解する。スキルの手順書を参照しなくても、対話の中で適切な問いを発せられるようになる。

### AI 側の蓄積

サイクルを回すたびに:
- ルール (`rules/`) にサイクルの原則が蒸留される
- メモリ (`memory/`) にフィードバックと判断履歴が蓄積される
- ADR (`docs/adr/`) に設計判断が記録される

これらがコンテキストとしてロードされるため、AI はスキルファイルを読まなくてもサイクルの原則に沿った行動を取れる。

### プラットフォーム側の吸収

ホストプラットフォーム自身が足場を吸収する: ツールやモデルがその能力をネイティブに搭載し、誰かが内面化したかどうかに関係なくローカルの artifact を冗長化する。前の 2 つのメカニズムと違い、これは外因性である — 練習ではなくプラットフォームのリリースサイクルが駆動する — そして現時点で 3 つの中の最速である。2026-06-10 の観測では、手組みのループから built-in 機能までのラグはおよそ 3 ヶ月だった。

### 補助輪のアナロジー

スキルは自転車の補助輪に相当する:
- 最初は補助輪がないと倒れる（スキルがないとサイクルが回らない）
- 乗れるようになったら補助輪は邪魔になる（スキルがコンテキストを消費するだけ）
- 補助輪を外しても自転車の乗り方は忘れない（サイクルの原則はルールとメモリに残る）

## What Survives: 情報差分の法則

溶解は単一の基準で artifact を選別する — 内容の高度さではなく、artifact とモデル/プラットフォームの既知との間の**情報差分**である:

- **差分ゼロ — 溶解する。** 普遍的な手順 (コードレビューのチェックリスト、TDD の手順、build-fix のループ) は訓練データであり、プラットフォームが真っ先に吸収する
- **差分 100% — 残る。** 個人の文脈 (自分の repo 群、voice 規約、公開パイプライン)、生きた外部サービスの操作 (DOI レジストリ、dataset mirror、knowledge graph エンドポイント)、そしてハーネス自身の自己管理には、訓練データの代替が存在しない

したがって汎用ハーネスは時間とともに、所有者の固有の実践のための operating system へと純化していく。「ニッチで異様」に見えるのは劣化ではない。汎用能力という水位の上昇が汎用的なものをすべて水没させたあとに残る姿である。

## 完了判定と溶解の順序

溶解は語りだけでなく観測の対象である。Measure フェーズが終了判定を供給する: ルールをロードした状態としていない状態で遵守計測 (skill-comply) を回し、両者が区別できなくなったら内面化は完了であり、ルールは削除してよい。規範文書が自らの死亡判定基準を内蔵している。

溶解の予測順序は、差分の法則を AKC 自身に適用すれば導ける:

1. **最初**: 一般的に記述された行動原則 (Research、Curate) — プラットフォームが吸収するものに最も近い
2. **最後**: Extract と Promote の内側にある判断閾値 — *何を残す価値があるか* は運用者の taste をエンコードしており、普遍解が存在しない

最後に残る一行は、ルールではなく好みである。

## Implications

1. **AKC の成功指標はスキルの使用率ではない** — 使われなくなることが成功
2. **スキルは永続的なツールではなく、一時的な教材** — 概念を伝える役割を終えたら退役してよい
3. **サイクルの持続性はスキルではなくルールとメモリに依存する** — スキルを削除してもサイクルは回り続ける
4. **新しいユーザー/AI ペアにはスキルが再び必要** — 足場は人に依存する。別の文脈では再構築が必要
5. **プラットフォーム吸収は全員の足場を一斉に退役させる** — 人に依存する内面化と違い、ホストプラットフォームがネイティブに搭載した能力は、以後のすべてのユーザー/AI ペアにとって足場を不要にする

## Relationship to Design Principles

この概念は AKC の既存の設計原則と整合する:

- **Composable** — 各スキルが独立しているから、個別に不要になれる
- **Non-destructive** — スキルを削除しても蓄積されたルール・メモリ・ADR は残る
- **Observable** — skill-comply で「スキルなしでもサイクルが回っているか」を検証できる
